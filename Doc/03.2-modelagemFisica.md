# Work Management / Time Tracking Template

# 03.2 - Dicionário de Dados

> Objetivo: Definir detalhadamente todas as tabelas e campos do sistema. Este documento representa o contrato oficial da estrutura do banco de dados e deverá ser utilizado como referência para geração das migrations e implementação do frontend.

---

# Filosofia

Cada tabela possui uma única responsabilidade.

Cada campo deve existir por um motivo.

Nenhum campo deve ser criado "caso precise futuramente".

Toda informação deve possuir uma origem bem definida.

---

# Estrutura Padrão

Todas as tabelas persistidas deverão possuir obrigatoriamente:

| Campo      | Tipo      | Obrigatório | Descrição               |
| ---------- | --------- | ----------- | ----------------------- |
| id         | UUID      | ✅          | Identificador único     |
| owner_id   | UUID      | ✅          | Identificador do tenant |
| created_at | Timestamp | ✅          | Data de criação         |
| updated_at | Timestamp | ✅          | Última atualização      |

Opcionalmente:

- deleted_at
- created_by
- updated_by

# Campos Calculados (Nunca Persistir)

Os seguintes dados devem ser sempre calculados:

- Horas trabalhadas
- Horas semanais
- Horas mensais
- Receita por cliente
- Receita por projeto
- Receita por colaborador
- Projeto mais ativo
- Cliente mais ativo
- Média diária
- Média semanal
- Horas faturáveis
- Horas não faturáveis

---

# Compatibilidade com o Tenant Gateway

Toda persistência deverá ser compatível com o CRUD genérico da MasIA.

Regras:

- Sem procedures.
- Sem triggers.
- Sem lógica SQL.
- Sem cálculos persistidos.
- Sem dependências de backend próprio.

Toda regra de negócio deverá existir na aplicação.
